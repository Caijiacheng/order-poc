## 2. 产品目标与范围

### 2.1 产品目标

本 POC 需要同时证明 3 件事：

1. 对经销商来说，它首先是一个标准 B2B 采购系统。
2. 对运营来说，它可以通过后台配置与批量生成机制，把“建议单”稳定地下发给经销商。
3. 对 IT 运维 / 演示保障来说，它具备可观测、可回滚、可追溯的运行闭环。

### 2.2 产品范围

本期只做：

- 面向经销商的前台采购链路
- 面向运营的后台配置与生成链路
- 面向 IT 运维 / 演示保障的观测、审计与回滚链路
- 推荐记录、批次记录、报表与 Langfuse 追踪
- 内存态数据与 mock 主数据

本期不做：

- 数据库持久化
- 真实 OMS / WMS / 库存 / 促销系统集成
- 生产级权限、安全、消息通知
- 复杂多 agent 自主决策框架

### 2.3 设计总原则

- 采购优先，AI 退居增强层
- 运营优先使用业务语言配置，不暴露开发者存储格式
- 推荐能力分成“后台批量预生成”和“下单实时优化”两类
- 决定性计算留在规则层，AI 负责解释、排序、结构化表达
- 技术观测与审计只在后台可见，不暴露给经销商前台

## 3. 用户角色与设计原则

### 3.1 核心角色

| 角色 | 核心目标 | 最关心什么 | 最怕什么 |
| --- | --- | --- | --- |
| 经销商采购员 | 今天该补的货下准、下快、一次过单 | 今日必补、热销、活动、门槛、箱规、账期、到货 | 缺货、压货、下单后才发现不合规、页面逼他研究 AI |
| 运营策略人员 | 把建议单推得准、推得稳、推得可控 | 给谁推、推什么、何时生成、是否发布、是否被采纳 | 配错策略、配错活动、批量覆盖错误、结果无法归因 |
| IT 运维 / 演示保障 | 保证演示稳定、问题可定位、配置可回滚 | 批次成功率、trace、配置基线、失败原因、兜底方案 | 客户面前无结果、trace 丢失、配置被改脏、无法恢复 |

### 3.2 角色边界

- 经销商只消费建议、调整购物车、确认订单。
- 运营只配置主数据、活动、策略、批量生成与发布，并复盘效果。
- IT 运维只保障运行、观测、审计和恢复，不参与经营策略内容判断。

### 3.3 关键设计约束

- 经销商前台禁止承担“生成建议”的职责。
- 运营后台禁止暴露 JSON 商品数组、逗号分隔主数据关联、内部 scene code 等技术结构。
- IT 运维后台必须能从“批次 -> 记录 -> trace -> 审计/回滚”快速定位问题。

### 4.3 两类推荐能力

#### A. 后台批量预生成建议单

用于：

- 常规补货
- 热销补货
- 活动备货
- 新品试销

特点：

- 由后台按天或按活动周期生成
- 前台直接消费已发布结果
- 经销商不触发“生成”

#### B. 下单实时优化建议

用于：

- 门槛补差
- 箱规修正
- 搭配补充

特点：

- 基于购物车状态自动计算
- 页面进入、加购、改量、删项后自动刷新
- 经销商只决定是否应用建议

## 6. 经销商视角 PRD

### 6.1 用户心智

经销商采购主线必须是：

`先看今天该补什么 -> 快速加入购物车 -> 补齐常规货 -> 系统给出凑单推荐 -> 确认交易条件并下单`

前台是标准 B2B 采购系统，不是 AI 功能演示页。

### 6.2 系统预生成 vs 用户主动操作

以下能力必须是系统预生成或自动刷新，不得由经销商点击触发：

- 三类组货模板（热销补货 / 缺货补货 / 活动备货）
- 购物车凑单
- 贴近提交区的推荐条

经销商主动操作仅包括：

- 搜索、筛选、加购、改量、删除
- 点击模板/活动 CTA 快速下单（先加车再跳转结算页）
- 确认收货、配送、结算、发票、备注并提交订单

前台禁止出现：

- 任何生成类、重跑类前台主动作
- 暴露 JSON、Prompt、run id、trace id、候选集等技术信息

### 6.3 前台 Canonical Route Matrix

| 页面 | Canonical URL | 页面目标 | 默认状态 | 系统动作 | 用户动作 |
| --- | --- | --- | --- | --- | --- |
| 采购工作台 | `/purchase` | 在同一页完成三类组货模板、活动专区、商品选购和采购摘要 | 默认展示模板化建议；无发布结果时回退模板候选与活动空态 | 加载已发布模板结果并维护右侧采购摘要 | 模板快速下单、活动快速下单、搜索筛选加购 |
| 购物车提交页 | `/order-submit` | 完成购物车清单核对、推荐条应用、交易信息确认与提交 | 购物车非空即显示贴近提交区的推荐条并带出交易信息 | 自动计算门槛补差/箱规修正/搭配补充，自动汇总订单金额与门槛状态 | 应用推荐条建议、改量删项、填写备注、提交订单 |

### 6.4 前台页面原型

#### 6.4.1 `/purchase` 采购工作台

必须包含：

- 三张组货模板卡（`热销补货`、`缺货补货`、`活动备货`）
- 活动专区
- 商品选购区
- 右侧采购摘要

模块说明：

- `三张组货模板卡`
  - 固定为 `热销补货`、`缺货补货`、`活动备货`
  - 每张卡都可直接加入购物车或快速下单
- `活动专区`
  - 展示可用活动、活动 SKU 与活动门槛
  - 支持活动快速下单
- `商品选购区`
  - 支持搜索、分类筛选、常购/待补货/活动/新品视图切换和加购
- `右侧采购摘要`
  - 展示 SKU 数、件数、金额、门槛差额
  - 提供 `去结算` 入口
  - 模板/活动的 `快速下单` 必须先加车，再跳转 `/order-submit`

默认状态：

- 经销商进入页面时，系统应已准备好该客户当天最新模板化建议结果
- 如无已发布建议单，页面回退显示三类模板候选与活动空态，不得出现自相矛盾标题

交互约束：

- “快速下单”语义固定为：点击模板或活动 CTA 后加入购物车并进入 `/order-submit`，不做静默直接提交
- 前台不出现任何“生成类”主动作

#### 6.4.2 `/order-submit` 购物车提交页

必须包含：

- 购物车清单（支持改量/删项）
- 贴近提交区的推荐条（金额补差、箱规修正、搭配补充）
- 收货信息
- 配送信息
- 结算方式 / 账期
- 发票信息
- 订单备注
- 订单汇总
- 提交订单

默认状态：

- 只要购物车非空，推荐条默认已计算完成
- 自动带出常用收货和结算信息

系统行为：

- 页面进入即自动计算
- 每次加购、删项、改量后自动重算
- 用户只点击“应用推荐”，不点击“生成建议”

### 6.5 AI 在前台的正确出现方式

AI 只回答 3 个业务问题：

- 为什么今天要补这批货
- 哪些货值得优先下
- 为什么这单还差一点、怎么补更合适

AI 的表现形式仅限于：

- 推荐理由标签
- 推荐原因抽屉
- 优化说明卡片
- 商品级建议标签

AI 不应表现为：

- 聊天主入口
- 生成按钮
- 技术链路跳转
- 模板或 Prompt 可视化

## 7. 运营策略视角 PRD

运营后台的核心工作流固定为：

`主数据与分群维护 -> 活动与推荐策略配置 -> 按天批量生成建议单 -> 结果发布 -> 效果复盘`

运营不是在后台“手写建议结果”，而是在后台配置“谁在什么场景下会收到什么类型的建议”。

### 7.3 运营页面原型

#### 7.3.1 运营工作台

必须展示：

- 今日批量生成结果
- 待处理异常
- 配置健康度
- 今日发布状态
- 快捷入口
- 最近变更

回答的核心问题：

- 今天建议单是否已经准备好
- 哪些经销商没有成功生成
- 哪些活动和策略今天正在生效

#### 7.3.2 主数据与分群

必须维护：

- 商品档案
- 经销商档案
- 经销商分群
- 商品池（常购池、热销池、新品池、活动池）
- 搭配关系

约束：

- 所有商品、活动、经销商关系都要通过结构化关联维护
- 禁止用逗号分隔文本维护 SKU 或客户类型关联

#### 7.3.3 策略与活动

必须维护：

- 活动策略
- 推荐策略
- 表达模板
- 全局规则

配置顺序应按运营心智组织：

`给谁 -> 推什么 -> 为什么 -> 何时生成 -> 如何发布`

而不是：

`scene code -> JSON -> prompt`

#### 7.3.4 生成与发布

必须支持：

- 生成前预检
- 抽样试生成
- 正式批量生成
- 生成后抽检
- 发布
- 撤回
- 补跑
- 过期管理

前台只消费“已发布且未过期”的建议单结果。

#### 7.3.5 数据复盘

必须支持按以下维度看效果：

- 时间
- 经销商
- 分群
- 策略类型
- 活动
- SKU
- 状态
- 采纳情况

### 7.4 推荐配置模型

#### 7.4.1 推荐策略类型

| 策略类型 | 执行方式 | 目标 | 关键字段 |
| --- | --- | --- | --- |
| 常规补货 | 按天批量 | 补齐日常基础货 | 经销商分群、常购池、补货触发周期、建议条数上限、建议量规则、排除条件 |
| 热销补货 | 按天/按周批量 | 强化高动销货 | 热销池、适用区域/渠道、热销等级、覆盖上限、建议量规则 |
| 活动备货 | 按档期批量 | 服务活动档期 | 关联活动、活动商品组、目标分群、备货窗口、活动优先级 |
| 新品试销 | 按周批量 | 控制新品试铺 | 新品池、适用经销商接受度、试销上限、观察期、止推条件 |
| 门槛补差 | 下单实时 | 接近门槛时补齐订单 | 触发差额区间、补差候选池、金额容忍度、低风险优先规则 |
| 箱规修正 | 下单实时 | 提升订单规范性 | 适用 SKU/品类、箱规接近阈值、修正方向、最大修正量 |
| 搭配补充 | 下单实时 | 补齐关联采购 | 来源商品组、目标商品组、搭配比、建议上限、排除规则 |

#### 7.4.2 表达模板

表达模板只负责：

- 理由怎么说
- 语气如何
- 重点强调什么
- 禁用词是什么

表达模板不负责：

- 决定给谁推
- 决定推哪些 SKU
- 决定数量逻辑

#### 7.4.3 全局规则

全局规则至少覆盖：

- 门槛金额
- 箱规倍数
- 起订量
- 搭配关系
- 禁推条件
- 替代关系
- 过期规则

### 7.5 配置控件真值

必须用结构化控件维护的对象：

- 经销商分群
- 商品池
- SKU
- 活动
- 活动商品
- 目标客户范围
- 搭配关系
- 排除项
- 优先级
- 状态
- 发布时间 / 过期时间

必须用“可增删行表格”维护的对象：

- 活动商品清单
- 搭配关系
- 排除项
- 参考建议项

可以保留自由文本的对象：

- 业务说明
- 运营备注
- 表达风格说明

禁止暴露给运营的内容：

- JSON 商品数组
- 逗号分隔 SKU / 客户类型
- 手填 `customer_id`
- 手填 `sku_id`
- 手填 scene code
- 底层 prompt JSON

### 7.6 批量生成与发布真值

生成流程固定为：

1. 选择业务日期、目标范围、策略包、发布方式
2. 系统做预检
3. 抽样试生成
4. 正式批量生成
5. 抽检结果
6. 发布或撤回
7. 复盘效果

建议单状态至少包括：

- 生成中
- 生成失败
- 待发布
- 已发布
- 已查看
- 已加购
- 已下单
- 已忽略
- 已过期

## 8. IT 运维 / 观测 / 演示保障视角 PRD

### 8.1 角色工作主线

IT 运维 / 演示保障的工作主线固定为：

`演示前检查 -> 演示中盯批次 -> 下钻记录与 trace -> 必要时回滚或重跑 -> 演示后复盘`

本角色不判断推荐“好不好”，只判断系统“稳不稳、能不能查、能不能恢复”。

### 8.2 运行观察对象分层

后台运行观察必须按以下层级组织：

- 运行工作台：现在能不能稳定演示
- 建议单批次：这次触发有没有成功产出
- 建议单记录：本次具体生成了什么
- Trace 链路：慢在哪、错在哪、断在哪
- 审计与回滚：谁改了什么、怎么恢复

关键约束：

- 运维入口必须先看“批次”，再看“记录”
- 推荐记录不是运维首页
- 后台批量生成任务与前台触发结果必须纳入同一批次视角

### 8.3 运行工作台要求

必须展示：

- 当前环境
- Demo 标识
- Langfuse 可用状态
- 最近一次成功推荐时间
- 最近 15 分钟推荐成功率
- 平均模型耗时
- 结构化输出失败次数
- 当前关键风险告警
- 当前基线配置是否被修改
- 最近批次
- 最近配置变更
- 快速回滚入口

### 8.4 建议单批次模型

系统需要显式承接 `recommendation_batch` 概念。

每个批次至少记录：

- batch_id
- batch_type
- trigger_source
- session_id
- customer_id
- scene
- trace_id
- related_run_ids
- config_snapshot_id
- started_at
- finished_at
- status
- error_summary
- fallback_used

批次状态至少包括：

- queued
- running
- success
- partial_failed
- failed
- cancelled
- fallback_served

### 8.5 推荐记录要求

推荐记录仍保留 run / item 两层，但在 IT 视角里属于批次下钻层。

run 级必须展示：

- recommendation_run_id
- batch_id
- trace_id
- strategy_id
- expression_template_id
- prompt_version / prompt_snapshot
- candidate_sku_ids
- returned_sku_ids
- model_latency
- status
- created_at / updated_at

item 级必须展示：

- recommendation_item_id
- sku_id / sku_name
- suggested_qty / rank
- action_type
- final_status
- was_explained
- was_applied
- applied_at
- rejected_reason

### 8.6 Trace 与 Langfuse 真值

顶级 trace 命名保持不变：

- `homepage.generate-recommendations`
- `cart.generate-optimization`
- `recommendation.explain`
- `admin.update-config`
- `confirm.submit-order`

关键 span 命名保持不变：

- `load.config-or-data`
- `rules.filter-or-calculate`
- `prompt.compose`
- `ai.invoke`
- `output.validate`
- `metrics.record`
- `apply-or-update`

约束：

- run 与 trace 必须双向可关联
- trace 缺失、span 缺失、run 和 trace 不一致都视为运行告警
- 观测埋点失败不得阻断主业务请求

### 8.7 告警、人工复核与回滚

后台必须能识别：

- 批次超时
- 批次部分失败
- 模板或策略缺失
- 候选 SKU 为空
- 结构化输出校验失败
- Langfuse 不可达
- run 与 trace 不一致
- 演示前短时间内发生关键配置改动
- 当前运行态偏离演示基线

人工复核入口必须支持：

- 重跑当前批次
- 用最近稳定配置重跑
- 打开 prompt snapshot
- 查看 candidate 与 returned SKU 差异
- 回滚指定配置对象
- 回滚到演示基线
- 重启应用恢复 seed
- 回放最近一次成功结果作为演示兜底

### 8.8 信息边界

仅允许后台可见：

- batch_id / run_id / trace_id / session_id
- Langfuse URL
- prompt_snapshot / prompt_version
- strategy_id / expression_template_id
- candidate_sku_ids / returned_sku_ids
- 模型名、延迟、token、span、错误码
- 配置 diff、回滚入口、基线快照、fallback 标记

绝不能暴露给经销商前台：

- 审计日志
- Trace ID 与 Langfuse 跳转
- Prompt 原文
- 配置版本与回滚信息
- 内部异常与失败详情
- 候选集与内部批次状态
- 人工重跑、回放、回退入口

## 9. 数据、AI、报表与内存态真值

### 9.1 内存态约束

- 所有主数据、策略、批次、记录、审计均为内存态
- 应用重启后恢复 seed 默认值
- 演示中允许临时修改配置，但必须可回滚或重启恢复

### 9.2 Seed 数据要求

Seed 至少覆盖：

- 商品
- 经销商
- 经销商分群
- 商品池 / 搭配关系
- 活动
- 推荐策略
- 表达模板
- 全局规则
- UI 配置

运行态必须额外沉淀：

- 批量生成任务
- 建议单批次
- 建议单
- 建议项
- 推荐记录
- 审计日志
- 运行 KPI

### 9.3 AI 真值

AI 只负责：

- 推荐结果排序
- 推荐理由生成
- 结构化建议表达
- 下单优化说明

AI 不负责：

- 计算门槛差额
- 计算总金额
- 计算箱规倍数
- 决定起订量
- 替代规则和禁推规则

统一服务入口仍为：

- `generateRecommendation()`
- `generateCartOptimization()`
- `generateExplanation()`

### 9.4 报表层次

报表必须分 4 层：

1. 聚合 KPI
2. 建议单批次
3. 建议单记录（run / item）
4. Trace / 审计

建议单记录报表至少支持以下筛选：

- 时间范围
- 经销商
- 分群
- 策略类型
- 活动
- SKU
- 状态
- 采纳情况

### 9.5 建议单与推荐记录的关系

业务对象关系为：

`推荐策略/活动 -> 批量生成任务 -> 建议单批次 -> 建议单 -> 建议项 -> 经销商采纳行为 -> 推荐记录与报表`

前台看到的是：

- 已发布建议单
- 实时优化建议

后台复盘看到的是：

- 建议单批次
- 建议单记录
- Trace
- 审计与恢复信息

## 10. 页面原型与交互验收基线

### 10.1 经销商前台验收

- 经销商进入采购工作台即可开始做单，不需要点击任何“生成”按钮
- 采购工作台内的商品选购区可在常购、待补货、活动、新品中快速切换
- 购物车提交页在改量、删项、加项后自动刷新优化建议
- 购物车提交页首先是交易确认页，其次才是优化说明页

### 10.2 运营后台验收

- 运营可以按业务语言配置活动、策略、表达模板和全局规则
- 运营可以发起预检、试生成、正式生成、发布、撤回和补跑
- 运营可以按批次、经销商、策略类型、活动和 SKU 复盘结果
- 后台不向运营暴露 JSON 商品数组和逗号文本主数据关联

### 10.3 IT 运维验收

- IT 可在后台直接看当前演示健康度
- 任一异常可沿“批次 -> 记录 -> trace -> 审计/回滚”定位
- Langfuse 不可达时业务不被阻断，但后台会告警
- 应用重启可作为最终 seed 恢复兜底
