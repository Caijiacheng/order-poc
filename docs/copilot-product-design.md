# AI 下单助手 Copilot 产品设计补充稿

## 1. 文档定位

本文件是对当前 POC 主 PRD 的专题补充，聚焦经销商前台的 `Copilot 自主填单能力`。

- 本文件不替代 [docs/product-design.md](/Users/caijiacheng/.codex/worktrees/4fbe/order-poc/docs/product-design.md)。
- 本文件用于说明 `采购页内嵌式 AI 下单助手` 的产品目标、页面形态、自治流程、活动补齐逻辑、核心运营指标与实施范围。
- 如果本文件与主 PRD 冲突，应以“采购优先、AI 退居增强层、决定性计算留在规则层”为更高原则，并在主 PRD 中补充正式表述。

本文件解决的问题是：

- Copilot 应该放在哪里，如何出现，如何不破坏标准 B2B 采购心智
- Copilot 的“自主”到底包括哪些动作，哪些不能自动做
- 活动补齐、门槛补差、采购预览与确认写车应该如何协同
- 运营后台应该先看哪些核心指标，而不是一开始就做过多复杂指标

## 2. 现状诊断

### 2.1 当前工程已经具备的基础

- 前台 canonical 路由已存在：`/purchase` 和 `/order-submit`
- `/purchase` 已具备三类组货模板、活动专区、商品选购区、右侧采购摘要
- `/order-submit` 已具备购物车核对、自动凑单、箱规修正、搭配补充、提交订单
- 购物车采用 session + 内存态模型，读写路径已完整
- 推荐生成、解释生成、购物车优化已接入统一 AI service，且具备结构化输出校验
- recommendation run、recommendation item、Langfuse trace、后台分析页已具备基础能力

### 2.2 当前工程尚未具备的能力

- 没有右下角 Copilot 入口，没有右侧抽屉式助手面板
- 没有 `一键做单` 的自治编排器
- 没有 “识别活动 -> 计算差额 -> 挑选合法补差组合 -> 生成采购预览 -> 用户确认写车 -> 再优化” 的可视化过程
- 没有 Copilot 专属 run / job / step 数据结构
- 没有面向 Copilot 的运营指标和后台漏斗
- 没有商品别名、俗称、模糊匹配的稳健机制

### 2.3 当前 PRD 的冲突点

主 PRD 当前写法倾向于限制前台 AI 不表现为聊天主入口。这个原则仍然正确，但需要补充说明：

- Copilot 不是全站聊天主入口
- Copilot 是采购页内嵌式助手
- Copilot 的职责是帮助经销商更快完成当前订单，而不是提供独立闲聊能力

建议后续把主 PRD 中的相关表述修订为：

- 允许在 `/purchase` 与 `/order-submit` 中存在 `可收起的内嵌式 AI 助手`
- 该助手必须服务当前采购单，不得脱离订单上下文
- 助手的主动作是 `辅助做单`，不是 `聊天`

## 3. 产品目标

Copilot 在本 POC 中需要证明 4 件事：

1. 经销商可以在标准采购页中，通过一个低打扰的入口，快速完成自主填单
2. Copilot 可以识别活动上下文与当前订单状态，对活动差额和补齐动作给出受控自治
3. Copilot 的过程是可解释、可观察、可追踪的，并且遵循“先预览、后确认写车”
4. 运营可以看到 Copilot 是否被使用、是否真的帮助做单、是否提高活动补齐和结算转化

## 4. 产品范围与非目标

### 4.1 本期范围

- `/purchase` 页面右下角 Copilot 入口与右侧抽屉
- `一键做单`
- `活动补齐`
- `解释这单`
- 文本偏好输入、快捷偏好输入、图片上传输入
- `生成采购预览`
- 用户确认后写入购物车
- `再跑一轮凑单 / 箱规优化`
- 面向运营的最小化 Copilot 指标

### 4.2 明确不做

- 全站级聊天中心
- 独立 thread list、多会话聊天管理
- 独立“图片下单中心”或独立 OCR 工作流
- 自动提交订单
- 真实 OMS / 促销系统 / 库存系统打通
- 高复杂度多 agent 框架
- 过多细颗粒度经营指标与复杂 BI 模型

## 5. 设计原则

- 采购优先：Copilot 必须服务采购动作，而不是抢占采购页面
- 真值单一：当前订单状态以当前 cart session 为准，不以聊天历史为准
- 事实优先：默认态只展示当前采购事实，不展示无来源的“AI 结论”
- 输入统一：文字、快捷偏好、上传图片都只是输入方式，不形成独立业务流
- 受控自治：模型不能自由发挥选货，只能在规则层给出的合法候选内做选择
- 过程可见：用户必须能看到“系统正在做什么”，但不能暴露技术日志和思维链
- 风险可控：低置信度、商品歧义、新品引入、超预算扩单等情况必须停下来确认
- 文案克制：前台文案使用采购业务语言，不使用拟人化、技术化、自说自话的表达
- 运营可复盘：关键动作、结果、失败原因要有 run 记录、事件埋点和 trace

## 6. 目标用户与主要任务

### 6.1 核心用户

| 角色 | 核心目标 | 使用 Copilot 的典型时机 |
| --- | --- | --- |
| 经销商采购员 | 更快地下准一单 | 打开采购页后想快速起单，或当前单还差活动门槛、采购门槛、箱规修正时 |
| 运营策略人员 | 看 Copilot 是否真的帮助做单 | 观察使用率、加车转化、活动补齐与结算转化 |
| IT / 演示保障 | 保证演示稳定 | 排查 Copilot run 是否成功、哪里失败、是否 fallback |

### 6.2 典型用户问题

- 帮我按今天活动和常购做一单
- 预算 6000 左右，优先补常购，不带新品
- 这单活动还差多少，帮我补齐
- 为什么加了这些货，能不能保守一点
- 把这张采购截图整理成一版单
- 现在这单能不能直接去结算

## 7. 产品形态

### 7.1 入口形态

- 页面位置：`/purchase` 与 `/order-submit` 右下角浮动入口
- 默认状态：收起，仅显示圆形 icon
- 点击后：从右侧展开助手抽屉
- 桌面端：抽屉宽度建议 `380-420px`
- 移动端：使用全屏或底部弹层

### 7.2 为什么采用右下角入口

- 不破坏当前采购页的主信息架构
- 不把 AI 变成页面中心
- 用户需要时可以主动打开，不需要时完全不打扰
- 形态上与截图参考一致，适合表达“随手可用的下单助手”

### 7.3 页面层级

- 第一层：采购页主内容
- 第二层：Copilot 入口
- 第三层：Copilot 抽屉
- 第四层：结果卡 / 阻塞卡 / 进度卡

Copilot 不应遮挡模板卡、活动区、商品区的主要操作，也不应改变主采购路径。

### 7.4 推荐的信息架构

`/purchase` 中的 Copilot 抽屉建议固定为以下顺序：

1. 顶部信息区
2. 当前采购情况区
3. 快捷动作区
4. 输入区（文本 + 快捷偏好 + 图片）
5. 过程可视化区
6. 结果区

设计要求：

- 默认态先展示事实状态，不展示推断式建议
- 用户先看到“当前单子是什么状态、能做什么”，再看到输入区
- 结果区只在触发动作后承担主视觉焦点
- 不把大聊天框作为默认主角

## 8. `/purchase` 页面中的 Copilot 设计

### 8.1 顶部信息区

- 当前经销商名称
- 当前订单金额 / SKU 数 / 当前活动状态
- 顶部状态以 chip 方式展示，推荐包括：
  - `当前金额`
  - `已选 SKU 数`
  - `本周可参与活动数`
- 活动状态 chip 不直接承担复杂交互，建议在旁边配轻链接 `查看活动`
- 顶部次级提示只展示事实状态，例如：
  - `还差 ¥800 达到起订额`
  - `已达到起订额，可继续去结算`

### 8.2 当前采购情况区

该区块是默认态下的事实摘要，不是“AI 结论”。

推荐展示：

- `已加载 3 组进货模板`
- `本周可参与活动 1 个`
- `可从模板、活动或商品区继续加购`

交互原则：

- 信息块以状态摘要为主
- 只允许少量轻链接，例如 `查看模板`、`查看活动`
- 不应在此区块展示无来源的建议结论，例如“建议先补热销再冲活动”

### 8.3 快捷动作区

- `一键做单`
- `活动补齐`
- `解释这单`

这些动作不是普通按钮堆叠，而是助手面板中的主要工作模式入口。

动作说明建议使用业务化文案：

- `一键做单`：按当前门店、活动和采购清单生成一版采购预览
- `活动补齐`：围绕活动门槛生成一版活动补齐预览
- `解释这单`：查看本次推荐或补齐的依据

注意：

- `一键做单` 不应依赖用户先输入文字
- 用户即使没有输入，也可以直接点击 `一键做单`
- 图片输入不单独成为第四种工作模式，而是作为补充输入方式并入当前三种动作

### 8.4 输入区（文本 + 快捷偏好 + 图片）

输入区用于承接用户偏好和补充约束，不承担订单真值。

适合表达的信息：

- 预算偏好
- 是否优先活动
- 是否只补常购
- 是否不要新品
- 是否偏保守

不适合表达的信息：

- 当前订单 SKU 真值
- 当前订单金额真值
- 当前活动差额真值

这些必须从当前 cart 和页面数据重新读取。

输入区推荐包含三部分：

1. 快捷偏好词
- `预算 6000 左右`
- `优先活动`
- `不要新品`
- `只补常购`
- `保守一点`

2. 文本输入框
- 用于补充本次偏好
- placeholder 使用业务化示例，例如 `比如：预算 6000，优先活动，不要新品`

3. 图片上传能力
- 支持上传采购截图、聊天截图、纸质订货单照片
- 图片上传只是“输入方式”，不是独立工作流
- 上传后在输入区下方展示轻量识别摘要，例如：
  - `已识别 8 行内容，可用于本次做单`
  - `其中 1 行待确认，1 行未匹配`
- 允许提供一个轻链接 `查看识别内容`

图片输入边界：

- 图片解析结果只是本次做单线索，不直接覆盖当前 cart 真值
- 图片上传建议仅在 `/purchase` 页面支持
- `/order-submit` 页面不承担图片起单职责

### 8.5 过程可视化区

当用户触发 `一键做单` 或 `活动补齐` 时，面板进入过程态，显示步骤卡。

建议固定展示以下步骤：

1. 读取当前订单与活动上下文
2. 如有图片，解析图片中的采购线索
3. 识别当前活动是否命中
4. 计算活动差额或订单缺口
5. 生成合法补差 / 补货组合
6. 生成采购预览
7. 用户确认后写入采购清单
8. 再跑一轮凑单 / 箱规优化

每一步有 4 种状态：

- 等待中
- 进行中
- 已完成
- 需确认 / 失败

用户看到的应该是业务过程，不是技术链路。

推荐展示：

- `已读取当前采购情况`
- `已识别图片中的 8 行采购条目`
- `其中 1 行待确认，1 行未匹配`
- `已从可补货商品中选出 1 组可执行组合`
- `已生成 1 版采购预览`
- `用户确认后已将 2 个 SKU 加入采购清单`

不应展示：

- Prompt
- 模型参数
- trace id
- 原始 JSON
- “OCR 工作流”“多模态调用”等实现术语

### 8.6 结果区

执行完成后先展示预览结果卡，用户确认后再进入已应用状态。

预览结果卡建议展示：

- 本次做单类型：`常规补货` / `活动备货` / `混合下单`
- 活动状态：`已命中` / `已补齐` / `未命中`
- 新增 SKU 数
- 订单金额
- 当前门槛状态
- 下一步动作：`加入采购清单` / `继续调整` / `去结算`

已应用状态建议展示：

- `已加入采购清单`
- `已同步最新采购金额与门槛状态`
- 若已满足条件，可继续引导去结算

结果区的默认空态应保持克制：

- `未生成采购预览，可先选择一种操作`

不建议在默认态展示：

- `本轮建议`
- `这版更稳`
- `我已为你分析`
- `建议先补热销再冲活动`

## 9. `/order-submit` 页面中的 Copilot 设计

`/order-submit` 不再承担“起单”职责，而是承担“交单前解释和收口”职责。

主要能力：

- 解释当前系统为什么给出门槛补差、箱规修正、搭配补充
- 继续帮用户处理“这单还差一点”的问题
- 判断当前订单是否适合直接提交

主要动作：

- `解释当前优化`
- `继续安全补齐`
- `现在去提交`

边界要求：

- `/order-submit` 不提供图片上传起单
- 若用户仍需通过图片整理采购内容，应返回 `/purchase`
- 结算页的 Copilot 只解释当前购物车与结算优化，不重新承担起单职责

## 10. 上下文模型

Copilot 必须按层次管理上下文。

### 10.1 一级真值：当前订单上下文

- 当前 session cart
- 当前 SKU 列表
- 当前数量
- 当前金额
- 当前门槛状态

规则：

- “这单”“当前这单”“现在这单” 都默认指当前 cart

### 10.2 二级真值：当前业务上下文

- 当前 dealer
- 当前页面
- 当前活动
- 当前模板 / 活动区 / 商品区数据

### 10.3 三级补充：用户输入上下文

- 预算偏好
- 常购优先
- 活动优先
- 不要新品
- 保守程度
- 用户附加的自由文本

### 10.4 四级补充：图片解析上下文

- 图片中识别出的条目行
- 识别条目的数量、单位、商品关键词
- 已匹配条目 / 待确认条目 / 未匹配条目

规则：

- 图片解析结果只是本次做单线索，不是订单真值
- 图片中的条目必须先经过匹配与校验，不能直接写入购物车
- 低置信度和商品歧义必须触发确认

### 10.5 五级临时态：Copilot 工作草稿

- 本次生成但未应用的草稿
- 仅用于面板内展示和回退
- 不覆盖 cart 真值

## 11. Copilot 能力定义

### 11.1 能力一：一键做单

适用场景：

- 用户从空购物车或半空购物车快速起一单
- 用户想按常购、活动、预算快速生成初稿

产出：

- 一个结构化采购预览草稿
- 用户确认写入后的真实购物车
- 一次二次优化后的结算建议

### 11.2 能力二：活动补齐

适用场景：

- 用户当前购物车已有活动品
- 用户想知道是否命中活动门槛
- 用户想让系统帮助整理一版补齐预览

产出：

- 当前活动金额
- 距门槛差额
- 合法补齐预览
- 用户确认后的加车结果

### 11.3 能力三：解释与微调

适用场景：

- 用户看不懂为什么推荐这些货
- 用户想让方案更保守或更偏活动

产出：

- 推荐原因说明
- 微调后的草稿或建议

### 11.4 输入方式定义

Copilot 本期只有三种主能力：

- `一键做单`
- `活动补齐`
- `解释这单`

文字、快捷偏好、上传图片都只是输入方式，不额外定义为独立能力。

推荐支持三种输入场景：

1. 只有文字
- 直接走当前能力

2. 只有图片
- 系统默认理解为：按图片内容整理一版采购预览

3. 文字 + 图片
- 图片提供商品线索
- 文字提供预算、活动优先级、新品限制等约束
- 最终仍走同一条 Copilot 编排链路

## 12. 一键做单产品流程

### 12.1 流程定义

用户点击 `一键做单` 后，系统按以下步骤运行：

1. 读取当前 dealer、cart、活动、模板、商品候选
2. 如有图片，先解析图片中的采购条目线索
3. 合并文本输入、快捷偏好和图片线索
4. 解析用户偏好和限制条件
5. 识别活动命中状态
6. 在规则层收窄候选集
7. 生成合法补货 / 补差组合
8. 由模型在合法组合中选出最合适的一组
9. 生成采购预览
10. 用户确认后写入购物车
11. 再跑一轮凑单 / 箱规优化
12. 输出结果卡并引导去结算

### 12.2 用户可见内容

用户应该看到的是业务过程，不是技术日志。

正确展示：

- 已识别当前活动
- 当前活动品金额为 `¥4680`
- 距活动门槛还差 `¥320`
- 已识别图片中的 `8` 行采购内容
- 其中 `1` 行待确认，`1` 行未匹配
- 已从可补齐商品中选中 1 组安全组合
- 已生成 1 版采购预览
- 用户确认后已将 2 个 SKU 加入购物车
- 已补齐 1 个箱规项

不应展示：

- Prompt
- Tool 名称
- trace id
- 原始 JSON
- 模型思维链

### 12.3 图片输入的产品处理原则

图片输入是统一工作流中的可选步骤，而不是独立业务流。

规则：

- 用户上传图片后，仍然通过 `一键做单`、`活动补齐`、`解释这单` 触发主动作
- 不新增独立的“上传图片下单”主模式
- 图片上传后的识别摘要只做轻量展示，不单独展开成第二主页面
- `查看识别内容` 可以打开轻量明细层，但其目的是校验输入，不是启动另一条流程

## 13. 活动补齐产品策略

### 13.1 活动补齐的核心问题

Copilot 不能只展示“当前活动还差多少”，还必须给出受控自治能力。

Copilot 需要回答：

- 当前购物车是否命中活动
- 还差多少
- 用什么补更安全
- 是否已经生成补齐预览
- 用户确认后是否已应用补齐

### 13.2 候选优先级

补齐活动门槛时，候选来源按优先级排序：

1. 当前活动 SKU 本身
2. 活动允许的搭配 SKU
3. 常购且安全的辅助 SKU

### 13.3 允许生成可应用预览的条件

只有满足以下条件时，才允许生成“可应用”的采购预览；真正写入购物车仍需用户确认：

- 候选组合合法
- 箱规合法
- 未命中禁配
- 未明显超出预算
- 不需要引入高风险新品
- 商品识别置信度足够高

### 13.4 必须阻塞确认的情况

- 商品歧义
- 图片识别歧义或 OCR 条目无法稳定匹配商品
- 需要引入新品
- 补齐需要明显扩单
- 活动补齐只能通过不常购或高风险 SKU 完成

## 14. 自治边界

### 14.1 可以自主做

- 读取当前订单和活动上下文
- 在有图片时解析图片中的采购线索
- 判断活动命中与差额
- 在规则层生成合法候选组合
- 在合法候选里选择最优一组
- 生成采购预览
- 在用户确认后写入购物车
- 再跑一轮凑单 / 箱规优化

### 14.2 不能自主做

- 直接提交订单
- 自己计算活动门槛、采购门槛、箱规倍数
- 在全量商品中随意挑选
- 无视用户偏好和风险条件强行扩单

## 15. 运营指标设计

本期只保留一组最小化核心指标，避免指标过多、解释成本过高。

### 15.1 核心指标列表

1. `AI 助手触发次数`
- 定义：用户打开 AI 下单助手并触发有效动作的次数
- 目的：判断功能是否被真正使用

2. `一键做单发起数`
- 定义：点击 `一键做单` 或 `活动补齐` 的次数
- 目的：判断用户是否愿意把起单交给 Copilot

3. `采购预览生成成功率`
- 定义：成功生成采购预览的次数 / 发起次数
- 目的：判断 Copilot 是否稳定产出可用结果

4. `加入采购清单成功率`
- 定义：用户确认写入购物车成功的次数 / 预览生成成功次数
- 目的：判断 Copilot 预览是否真的被用户接受

5. `活动补齐成功率`
- 定义：识别到活动差额后成功补齐的次数 / 活动补齐发起次数
- 目的：衡量 Copilot 在活动经营上的实际价值

6. `AI 助手辅助结算转化率`
- 定义：AI 下单助手写车成功后，最终进入 `/order-submit` 并提交订单的比例
- 目的：衡量 AI 下单助手是否真的帮助成交

7. `AI 助手运行成功率`
- 定义：AI 下单助手 run 完成率
- 目的：衡量系统稳定性

8. `AI 助手平均耗时`
- 定义：从发起到结果卡完成的平均时长
- 目的：衡量体验是否足够快

### 15.2 暂不纳入首版的指标

- 精细化 SKU 删除率
- 人工改量率
- 各步骤详细漏斗
- 复杂策略贡献归因
- 多维度人群分层效果
- 图片识别专属 BI 大屏

这些可在第二阶段基于基础事件继续扩展。

## 18. 技术形态建议

本文件是产品设计文档，但为避免方案失真，需明确技术承载方式。

- 前端容器建议使用 `assistant-ui`
- 对话运行时建议使用 AI SDK v6 集成
- 一键做单建议使用服务端 orchestrator，而不是把所有自治流程都塞进聊天文本
- 图片上传建议复用 `assistant-ui` / AI SDK 的附件消息能力
- 图片识别建议通过单轮 OCR 模型作为工作流子步骤接入，而不是做成独立产品流程
- 规则层继续承担门槛、箱规、活动差额、禁配等决定性计算
- 模型只负责：
  - 解析意图
  - 在有图片时提取图片中的采购条目线索
  - 在合法候选中排序与选择
  - 生成业务解释

### 18.1 Copilot 的 LLM 设计原则

Copilot 的 LLM 设计不是“只写一段提示词”，也不是“完全靠复杂 agent 框架”。

推荐设计是：

- `提示词` 负责约束模型角色、输出格式与边界
- `自主工作流` 负责业务步骤编排
- `规则层` 负责金额、门槛、箱规、活动差额、禁配等决定性计算
- `模型` 只负责意图理解、候选排序与业务表达
- `OCR 子步骤` 只负责把图片转成结构化条目线索，不直接生成订单真值

一句话原则：

- 让工作流控制订单
- 让提示词约束模型
- 让模型参与判断，但不直接控制最终订单真值

### 18.2 为什么不能只靠提示词

如果 Copilot 只依赖提示词直接生成订单，首版 POC 会出现以下问题：

- 同一句需求多次生成结果不稳定
- 模型可能误算活动门槛、采购门槛、箱规倍数
- 商品歧义、活动范围、禁配规则难以被严格执行
- 运营和 IT 无法清楚知道系统到底在哪一步出错

因此必须采用：

- `提示词 + 受控工作流`

其中：

- 问答解释类能力可以更偏提示词驱动
- 一键做单和活动补齐类能力必须偏工作流驱动

### 18.3 Copilot 中适合由 LLM 负责的任务

以下任务适合由模型承担：

- 从用户自然语言中提取偏好
  - 预算
  - 活动优先
  - 常购优先
  - 不要新品
  - 保守程度
- 在规则层已生成的合法候选组合中选择最合适的一组
- 用业务语言解释为什么这样补货、为什么这样补活动差额
- 在阻塞场景下把系统问题翻译成用户听得懂的追问

### 18.4 Copilot 中不适合由 LLM 负责的任务

以下任务不应交给模型直接决定：

- 自己计算活动门槛差额
- 自己计算采购门槛差额
- 自己计算箱规倍数
- 在全量商品中任意搜索并自由下结论
- 在用户未确认前直接写入购物车
- 直接提交订单

这些动作必须由代码和规则层完成。

### 18.5 推荐的工作流结构

Copilot 的“一键做单”建议采用显式服务端工作流，而不是单轮聊天直接出结果。

推荐步骤：

1. `load_context`
2. `image_extract`（可选）
3. `parse_intent`
4. `detect_campaign_state`
5. `build_legal_candidates`
6. `select_best_combo`
7. `build_preview`
8. `apply_to_cart`（用户确认后）
9. `run_cart_optimization`
10. `summarize_result`

各步骤分工如下：

- `load_context`
  - 由代码完成
  - 读取当前 cart、dealer、活动、模板、商品候选
- `image_extract`
  - 由代码调用 OCR 模型完成
  - 将图片转成条目线索、识别状态和待确认项
- `parse_intent`
  - 由模型完成
  - 将用户自然语言与图片条目线索转成结构化偏好
- `detect_campaign_state`
  - 由代码完成
  - 判断当前是否命中活动、当前活动金额、活动差额
- `build_legal_candidates`
  - 由代码完成
  - 按规则生成合法补差 / 补货候选组合
- `select_best_combo`
  - 由模型完成
  - 在合法候选中选最合适的一组
- `build_preview`
  - 由代码完成
  - 将选择结果整理成用户可确认的采购预览
- `apply_to_cart`
  - 由代码完成
  - 在用户确认后调用现有购物车 service 写入购物车
- `run_cart_optimization`
  - 由代码完成
  - 调用现有凑单 / 箱规 / 搭配优化逻辑
- `summarize_result`
  - 可由模型或模板完成
  - 输出用户可读的结果说明

### 18.6 推荐的提示词分层

Copilot 不建议只有一个超长提示词。建议至少拆成三层：

- `系统提示词`
  - 定义角色、边界、禁止事项、输出原则
- `步骤提示词`
  - 针对 `parse_intent`、`select_best_combo`、`summarize_result` 等具体任务分别编写
- `动态上下文注入`
  - 注入当前 cart、dealer、campaign、候选组合、用户偏好

### 18.7 系统提示词应该写什么

系统提示词的目标不是教模型“怎么计算”，而是明确边界。

必须覆盖以下内容：

- 你是采购页内嵌式 AI 下单助手，不是通用聊天机器人
- 你服务的是当前经销商、当前订单
- 你不能把聊天内容当作订单真值，订单真值始终以当前 cart 为准
- 你不能自行计算门槛、箱规、活动差额
- 你只能在提供给你的合法候选组合中做选择
- 如果商品识别不清、需要引入新品、扩单风险高，必须中断并请求确认
- 你输出的结果必须是结构化 JSON，不要输出额外解释文字

建议系统提示词骨架如下：

```text
你是一个面向经销商采购页的 AI 下单助手。

你的职责：
1. 理解用户当前的采购意图和偏好。
2. 基于系统提供的当前订单上下文、活动上下文和合法候选组合，选择最合适的方案。
3. 用业务语言说明方案理由。

你的边界：
1. 当前订单真值永远以系统提供的 cart 为准，不以聊天历史为准。
2. 不要自行计算采购门槛、活动门槛、箱规倍数、价格和差额。
3. 只能在系统提供的合法候选组合中做选择，不能创造新的 SKU 或新的组合。
4. 遇到商品歧义、新品引入、明显扩单、风险过高时，必须返回需要确认的阻塞结果。
5. 输出必须符合指定 schema，不要输出 schema 之外的文本。
```

### 18.8 `parse_intent` 提示词应该怎么写

这个步骤的目标是把用户输入转成结构化偏好，而不是生成订单。

当存在图片输入时，应把已结构化的图片条目线索一并提供给该步骤，但仍不得把图片识别结果当作订单真值。

建议抽取字段：

- `intent_type`
- `budget_target`
- `prefer_campaign`
- `prefer_frequent_items`
- `avoid_new_products`
- `risk_mode`
- `must_have_keywords`
- `exclude_keywords`

建议提示词骨架如下：

```text
请从用户输入中提取采购偏好，不要生成订单，不要推断不存在的商品。

输出要求：
- 返回结构化 JSON
- 如果用户没有明确说明预算、活动优先、是否不要新品，则保留为空或默认值
- 如果用户表达模糊，不要擅自补全为强约束

你要提取的字段包括：
- intent_type: start_order | topup_campaign | explain_order | adjust_order
- budget_target: number | null
- prefer_campaign: boolean | null
- prefer_frequent_items: boolean | null
- avoid_new_products: boolean | null
- risk_mode: conservative | balanced | aggressive | null
- must_have_keywords: string[]
- exclude_keywords: string[]
```

### 18.9 `select_best_combo` 提示词应该怎么写

这是 Copilot 中最关键的模型步骤之一。

这里的提示词重点不是“让模型自己找货”，而是强调：

- 候选组合已经是合法组合
- 你只能从这些组合中选
- 选择标准是更贴近当前 dealer 和当前用户偏好

建议提示词骨架如下：

```text
系统已提供当前经销商信息、当前订单状态、用户偏好，以及若干个已经过规则校验的合法候选组合。

你的任务：
- 只能从这些候选组合中选择一个最合适的组合
- 不要创建新组合
- 不要新增系统未提供的 SKU
- 选择时优先考虑：
  1. 是否满足用户表达的偏好
  2. 是否更贴近活动补齐目标
  3. 是否更符合常购和稳健采购倾向
  4. 是否避免引入高风险新品

输出要求：
- 返回 combo_id
- 返回业务解释 explanation
- 如果没有任何安全组合满足要求，返回 blocked，并给出阻塞原因
```

### 18.10 `summarize_result` 提示词应该怎么写

这个步骤的目标是把执行结果翻译成用户听得懂的话。

重点要求：

- 用经销商采购语言，不用技术语言
- 不要暴露 trace、run id、schema、prompt
- 聚焦结果：补了什么、为什么、现在是否适合去结算

建议提示词骨架如下：

```text
请把系统执行结果整理成采购员能快速理解的结果说明。

要求：
- 语言简洁、直接、业务化
- 不要出现技术术语、链路信息、模型信息
- 优先说明：
  1. 当前活动是否命中
  2. 补了哪些货
  3. 当前金额和门槛状态
  4. 是否建议去结算
```

### 18.11 提示词中必须禁止的内容

以下内容建议写进系统级约束：

- 禁止自行计算金额、差额、箱规
- 禁止编造 SKU
- 禁止在无候选时强行给出推荐
- 禁止把用户上一轮聊天结果当作当前订单真值
- 禁止输出未要求的自然语言长段落
- 禁止越过确认直接引入高风险新品或大幅扩单

### 18.12 输出形式建议

Copilot 所有关键 LLM 步骤都建议走结构化输出，不建议返回自由文本再解析。

推荐方式：

- `parse_intent`：结构化 object
- `select_best_combo`：结构化 object
- `summarize_result`：结构化 object 或短文本字段

不建议：

- 让模型直接输出“我建议加 A、B、C”
- 再由前端或后端用正则、字符串切分去猜它的意思

### 18.13 首版落地建议

首版不要追求一个万能 Copilot 提示词。

建议首版只落 3 套关键提示词：

1. `parse_intent`
2. `select_best_combo`
3. `summarize_result`

其他步骤全部先走代码和规则层。
