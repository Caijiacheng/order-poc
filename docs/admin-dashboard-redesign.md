# 后台 IA 与 Dashboard 重构

## 1. 目标

将当前后台从“平铺 demo 配置页集合”重构为标准的一、二级菜单后台控制台。

目标不是把页面变多，而是让后台看起来像：

- 标准 B2B 运营配置台
- 有清晰领域分组的一、二级菜单
- 有明确职责的 dashboard
- 配置、报表、审计、演示观察彼此分层

---

## 2. 我对当前后台的诊断

当前后台最大的问题不是页面少，而是“没有后台语言”，更像一组平铺链接。

### 2.1 现在的导航是页面清单，不是后台 IA

当前 `ADMIN_NAV` 是一个平铺数组：

- Dashboard
- Products
- Dealers
- Templates
- Campaigns
- Rules
- Prompts
- Reports
- Reco Records

这更像开发阶段的路由列表，不像一个控制台的一、二级菜单体系【F:/Users/caijiacheng/AIProject/order-poc/lib/navigation.ts†L8-L17】。

### 2.2 一级菜单没有领域语义

标准后台的一级菜单通常按“业务领域”划分，例如：

- 工作台
- 主数据
- 策略配置
- 经营分析
- 系统与审计

当前后台直接把每个页面都放成一级入口，用户无法快速理解：

- 哪些是基础档案
- 哪些是推荐策略
- 哪些是活动配置
- 哪些是报表
- 哪些只是观察链路

### 2.3 Dashboard 只是模糊总览，不是后台首页

当前 dashboard 主要展示：

- 商品数
- 经销商数
- 模板数
- 活动数
- 运行 KPI
- 推荐批次状态
- 最近审计日志

这些信息本身没问题，但它没有形成一个后台首页应有的结构。它更像把几块数据卡片放在第一页，而不是一个有“今日状态 + 风险提醒 + 快捷入口 + 最近变更 + 关键结果”的工作台【F:/Users/caijiacheng/AIProject/order-poc/app/admin/dashboard/page.tsx†L68-L222】。

### 2.4 配置页和报表页没有清晰分层

当前导航上：

- `Products / Dealers / Templates / Campaigns / Rules / Prompts`
- `Reports / Reco Records`

它们混在同一层级，导致后台看起来不像“运营配置台”，而像“接口功能目录”。

### 2.5 后台风格不统一

当前 admin shell 是一个左侧扁平导航 + 顶部小标题：

- 左侧像链接盒子
- 顶部像阶段说明条
- 中间内容是页面区块

这更接近 demo 控制面板，而不是成熟后台产品【F:/Users/caijiacheng/AIProject/order-poc/app/admin/layout.tsx†L9-L66】。

### 2.6 “返回演示台”暴露过早

后台顶部把 `返回演示台` 放得很显眼，这会强化“这是 demo 附页”的感觉，而不是“这是一个独立后台控制台”【F:/Users/caijiacheng/AIProject/order-poc/app/admin/layout.tsx†L47-L58】。

演示跳转应该保留，但不应压过后台本身的信息架构。

## 3. 重构原则

### 3.1 一级菜单按领域分组

一级菜单负责表达“这是哪类工作”。

### 3.2 二级菜单按页面落地

二级菜单负责表达“这一类工作里具体能做什么”。

### 3.3 Dashboard 是工作台，不是数据拼盘

后台首页必须回答 4 个问题：

- 当前系统今天运行得怎么样
- 哪些配置最值得关注
- 最近发生了什么变更
- 我下一步最可能要去哪里

### 3.4 配置、分析、审计、演示观察要分层

不要把“改配置”和“看结果”和“查链路”混在同一层。

### 3.5 命名用业务语言，不用技术路由语言

后台用户更容易理解：

- 商品档案
- 经销商档案
- 推荐模板
- 活动策略
- 推荐规则
- 推荐记录
- 审计日志

而不是：

- Products
- Dealers
- Reco Records

### 3.6 演示属性要弱化，不要消失

这是 POC，所以仍要支持：

- 返回前台
- 看推荐链路
- 看 Langfuse

但这些能力应放在“演示观察”或“审计/链路”层，而不是主导航正中心。

## 4. 一级 / 二级菜单重构

我建议后台改成 5 个一级菜单。

### 4.1 一级菜单：工作台

角色：

- 运营负责人 / 演示负责人进入后台后的首页

二级菜单：

- `运营总览`
  - 对应当前 `/admin/dashboard`

职责：

- 看今日运行
- 看配置健康度
- 看推荐效果摘要
- 看最近变更
- 进入常用配置页

### 4.2 一级菜单：主数据管理

角色：

- 维护基础业务对象

二级菜单：

- `商品档案`
  - 对应 `/admin/products`
- `经销商档案`
  - 对应 `/admin/dealers`

职责：

- 管理商品主数据
- 管理经销商画像

### 4.3 一级菜单：推荐与策略

角色：

- 运营策略人员 / 产品经理

二级菜单：

- `推荐模板`
  - 对应 `/admin/suggestion-templates`
- `活动策略`
  - 对应 `/admin/campaigns`
- `推荐规则`
  - 对应 `/admin/rules`
- `Prompt 配置`
  - 对应 `/admin/prompts`

职责：

- 维护推荐样板
- 管理活动引导
- 配置业务规则
- 调整模型提示词

说明：

- 这是整个 POC 的“策略中心”

### 4.4 一级菜单：数据与分析

角色：

- 看结果、看采纳、看效果

二级菜单：

- `经营总览`
  - 对应 `/admin/reports`
- `推荐记录`
  - 对应 `/admin/reports/recommendations`

职责：

- 看聚合 KPI
- 看推荐 run 和 item 级记录
- 看采纳、金额提升、客户分布

### 4.5 一级菜单：审计与演示观察

角色：

- 演示负责人 / 调试人员 / 管理员

建议二级菜单：

- `变更日志`
  - 当前可先挂接审计日志视图
- `链路观察`
  - 可作为后续页，整合 Langfuse、trace、演示入口

说明：

- 这一组菜单是 POC 特有层
- 它不应压在主配置导航里，但也不应消失

## 5. 推荐的后台导航形态

建议形态不是“左侧一列按钮”，而是标准后台的：

- 左侧一级菜单
- 一级展开后的二级菜单
- 右侧内容区
- 顶部全局状态条

### 5.1 左侧导航

左侧第一层显示一级菜单：

- 工作台
- 主数据管理
- 推荐与策略
- 数据与分析
- 审计与演示观察

点中一级菜单后，展开对应二级菜单。

### 5.2 顶部条

顶部建议展示：

- 当前后台名称：`运营配置台`
- 当前环境：`POC / Demo`
- 最近更新时间
- 返回前台入口

但“返回前台”应是辅助入口，不应像现在这样成为顶部高权重动作。

### 5.3 面包屑

进入页面后建议显示：

- `推荐与策略 / 推荐规则`
- `主数据管理 / 商品档案`
- `数据与分析 / 推荐记录`

这会显著提升后台的可理解性。

## 6. Dashboard 应该怎么重构

当前 dashboard 更像“总数据页”，我建议改成真正的后台首页工作台。

### 6.1 Dashboard 的角色

后台首页应该服务于 3 件事：

1. 看今天系统运行状态
2. 快速进入最常用配置工作
3. 看最近是否有异常或重要变化

### 6.2 Dashboard 建议布局

#### 第一屏：今日运行状态

展示 4-6 张核心卡：

- 今日推荐批次数
- 今日采纳次数
- 采纳率
- 今日金额提升
- 平均模型耗时
- 结构化输出失败次数

#### 第二屏：配置健康度

展示配置对象完整度：

- 商品数 / 启用数
- 经销商数 / 启用数
- 推荐模板数 / 启用数
- 活动数 / 启用数
- 最近 7 天是否有配置修改

#### 第三屏：快捷工作入口

推荐做成“快捷卡片”：

- 新增商品
- 新增经销商
- 新建推荐模板
- 调整推荐规则
- 调整 Prompt
- 查看推荐记录

#### 第四屏：最近变更

展示：

- 最近 5-10 条审计日志
- 谁改了什么
- 修改时间

#### 第五屏：推荐效果摘要

展示：

- 推荐批次状态分布
- 各经销商推荐量分布
- 采纳状态分布
- 最近一次高价值 run

### 6.3 Dashboard 不该承担的内容

不建议在 dashboard 首页直接塞太多：

- 详细表格
- 大量原始记录
- Langfuse 链路明细
- 单条推荐 run 详情

这些应进入二级页面。

## 7. 前台与后台的分层建议

当前一个核心问题是：后台看起来像前台 demo 的附属页。

我建议明确分层：

### 7.1 前台

角色：

- 经销商采购体验

关键词：

- 采购
- 补货
- 选品
- 购物车
- 下单

### 7.2 后台

角色：

- 运营配置
- 策略维护
- 数据分析
- 审计观察

关键词：

- 档案
- 模板
- 规则
- 策略
- 报表
- 审计

换句话说：

- 前台像交易系统
- 后台像控制台

这两者的视觉和 IA 都应该拉开距离。

## 9. 最终判断

如果你希望后台一眼看上去就是“标准后台”，最关键的不是把卡片做得更漂亮，而是把结构改对：

1. 用一级菜单表达领域
2. 用二级菜单承接页面
3. 把 dashboard 从“总览页”改成“工作台”
4. 把配置、分析、审计、演示观察分开
5. 把 demo 属性降级成辅助层，而不是后台主气质

我建议你下一步如果继续落地代码，优先顺序应该是：

1. 先改后台 IA 与导航
2. 再改 dashboard 结构
3. 再统一后台命名和视觉语气
4. 最后再细化各页面内容

这样改完之后，后台会更像一个标准的 B2B 运营配置台，而不是 POC 附属控制面板。
